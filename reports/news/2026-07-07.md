<b>═ 🔥 TOP STORY ═</b>
Espressif's developer portal published a technical deep-dive this June on running Rust's smoltcp stack as the IPv4/IPv6 data plane inside ESP-IDF — a drop-in alternative to the C-based lwIP stack that ships by default. A linker `--wrap` shim keeps esp_http_server, esp-tls, and esp-mqtt working with no source changes, while smoltcp itself is a #[no_std], heap-free stack with a single explicit poll loop: no background thread quietly touching your buffers, no mutex-guarded multitasking the way lwIP juggles several internal tasks, and packet parsing checked by Rust's borrow checker instead of a hand-rolled bounds check. Espressif measured 91.15 Mbit/s on an ESP32-P4 over a 100 Mbit link — essentially wire-speed, so the memory-safety trade isn't costing throughput. This lands the same stretch of weeks this digest has repeatedly covered memory-corruption bugs in exactly this class of C code — FatFs, Zephyr's DNS resolver — and gives ESP32 developers a working, ESP-IDF-native way to remove that whole bug class from the network layer without rewriting the application on top of it.

<b>1. Post: Security wake-up — a 9.4-severity bug in Zephyr's DNS resolver</b>
A CVSS 9.4 out-of-bounds write in Zephyr RTOS's DNS parser needs nothing from an attacker except a crafted DNS response. No auth, no local access, no user click.

The bug is almost embarrassingly simple: dns_unpack_name() measures the buffer's remaining space once, at the start of parsing a name, and never rechecks it as it appends labels. Send a response with oversized labels and the function writes past the 255-byte name buffer it thought it still had room in.

What gets me is the trigger condition — this fires on ordinary DNS resolution, the kind of thing almost every network-connected embedded device does without a second thought. No packaged patched release exists yet either, just fix commits sitting on a few branches.

If you've shipped anything on Zephyr that resolves hostnames over an untrusted network, this is worth 20 minutes today.

<b>2. Post: Integration leap — Rust's smoltcp quietly becomes a real lwIP alternative on ESP32</b>
For over a decade, "TCP/IP on an ESP32" has meant lwIP: a fast, battle-tested, and very much C, multi-task network stack. Espressif just published a documented path to swap it for Rust's smoltcp instead — without rewriting your app.

A linker wrap shim keeps esp_http_server, esp-tls, and esp-mqtt compiling against the new stack untouched. smoltcp itself has no heap requirement, no background thread, and a single explicit poll loop — the code that parses bytes from the network, historically the most attacked surface in any embedded stack, is now safe Rust instead of hand-rolled C.

The number that got my attention: 91.15 Mbit/s measured on an ESP32-P4 over a 100 Mbit link. Near wire-speed. Memory safety here isn't costing you throughput, which is usually the argument against it.

Would you trade a small integration project for provably safer packet parsing?

<b>3. Post: Physics-meets-architecture — 30x more efficient AI inference without a new process node</b>
Ambiq's new Apollo510 claims a 30x power-efficiency jump for endpoint AI over its previous generation. No die shrink, no new fab node — just architecture.

The chip pairs an Arm Cortex-M55 with Helium vector extensions, clocked at 250MHz, but the real lever is Ambiq's "SPOT" platform — running logic at subthreshold voltage, below where transistors normally switch cleanly, and architecting around that instead of fighting it. The claimed payoff: roughly 10x lower inference latency and 2x lower energy per inference than Apollo4.

This is the part worth sitting with if you design for battery-constrained hardware: efficiency gains are increasingly coming from voltage-domain and micro-architecture decisions, not from waiting for the next process node — which matters a lot once you stop assuming Moore's Law does the work for you.

<b>4. Post: Tooling unlock — tracing your firmware without patching FreeRTOSConfig.h</b>
Every time a new trace library shows up for ESP-IDF, it used to mean hooking FreeRTOS events by hand and editing FreeRTOSConfig.h directly. Espressif's new esp_trace component ends that.

It splits application-level tracing into a small core plus two swappable pieces: an encoder that defines the trace format, and a transport that moves the bytes off-chip. Both are resolved from a registry by name at startup, so combining them — say, switching from JTAG to USB-Serial-JTAG as the transport — needs no source changes to the trace library itself.

The detail that matters for anyone who's fought this before: a stable extension point (CONFIG_ESP_TRACE_LIB_EXTERNAL) means a third-party trace recorder can plug in without touching ESP-IDF's core at all — the format and the wire are no longer welded together.

Anyone building custom instrumentation on ESP32 — was patching FreeRTOSConfig.h ever not painful?
