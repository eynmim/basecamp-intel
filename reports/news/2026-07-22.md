<b>═ 🔥 TOP STORY ═</b>
Zephyr's Bluetooth Audio Broadcast Assistant has shipped a classic embedded-C bug since version 3.6: the Broadcast Receive State reassembly logic uses a single file-static 512-byte buffer shared across every connected peripheral instead of a per-connection buffer. When a device acts as a Broadcast Assistant to multiple Scan Delegators at once — a normal Auracast/LE Audio use case — notifications from different connections interleave on that shared buffer, and with large ATT MTUs a malicious or compromised peripheral can push it into an out-of-bounds write, corrupting memory or mixing data across connections (GHSA-73c7-3rh7-v5p9, CVSS 6.4). The fix moves the buffer into per-connection instance storage and has already merged to Zephyr's main branch, but it only ships in the 4.5.0 release due in October 2026 — meaning every 4.4.0-and-earlier device is exposed for months unless a maintainer manually backports commit 0cd6158. Why it matters: this is the exact anti-pattern — one static buffer standing in for what should be per-instance state — that keeps resurfacing in RTOS network stacks, and it's a direct hit against anyone building multi-connection BLE Audio on Zephyr.

<b>1. Post: Security wake-up — one shared buffer, every Bluetooth connection at once</b>
A single 512-byte buffer, declared once, shared by every connection your device has open. That's the bug Zephyr just found in its own Bluetooth Audio stack.

The BAP Broadcast Assistant — the role a device plays when it's coordinating multiple Bluetooth LE Audio broadcasts — was reassembling incoming notifications from every connected peripheral into one file-static buffer instead of giving each connection its own. Run it against more than one peripheral at a time (a completely normal Auracast use case) and their data interleaves. Push a large enough ATT MTU through a malicious or compromised peripheral and you get an out-of-bounds write.

The fix is already merged, but it ships in Zephyr 4.5.0 in October — months from now. Every 4.4.0-and-earlier device stays exposed unless someone backports it by hand.

Static buffers standing in for per-instance state is one of the oldest bug classes in embedded C. It's still finding new ways in.

<b>3. Post: Tooling unlock — PyTorch to microcontroller just lost two conversion hops</b>
Getting a PyTorch model onto an MCU used to mean PyTorch → ONNX → TensorFlow → TensorFlow.js, hoping nothing broke at each hop.

Google's LiteRT Torch converts PyTorch models straight to .tflite in a single step. It ships alongside LiteRT.js, which runs those .tflite models natively in the browser via WebAssembly — XNNPACK on CPU, WebGPU or the experimental WebNN API when a GPU/NPU is available — replacing TensorFlow.js's older JS-only kernels entirely.

The practical effect: if your model started life in PyTorch (increasingly the default for anyone coming out of a research pipeline), the path to a TFLite Micro deployment on an ESP32-S3 or STM32N6 just got a lot shorter, with fewer places for numerical drift to sneak in during conversion.

Anyone still maintaining a multi-hop conversion pipeline who's ready to rip it out?
