<b>═ 🔥 TOP STORY ═</b>
ESPHome 2026.6.0 landed this month as the first release to fully prepare the ESPHome firmware ecosystem for ESP-IDF 6, and the engineering headline is native RISC-V clang-tidy support for ESP32-C3, C6, H2, and P4 targets. The old approach ran RISC-V code through a 32-bit x86 stand-in for the Xtensa fallback path — meaning static analysis silently missed RISC-V-specific alignment faults, pointer-width assumptions, and register-clobber bugs that only manifest on the actual target architecture. The new clang-tidy pass uses clang's upstream RISC-V backend, pins sdkconfig to newlib, and strips GCC-only flags, making static analysis genuinely architecture-accurate for the first time. Simultaneously, the release retires the legacy in-tree ESPHome dashboard (replaced by Device Builder 1.0), delivers a complete audio-stack modernisation (zero-copy ring buffers, any-bit-depth mixing, new router speaker), and makes enable_on_boot: false on Wi-Fi and Ethernet actually reclaim the 15–30 KB and 3–8 KB of internal SRAM they were always supposed to. For ESP32 firmware engineers building on RISC-V targets, this is the release that makes CI-integrated clang-tidy worthwhile — the old stub would have passed architecture-specific bugs straight through.

<b>2. Post: [Capability unlock] — One ESP32 die, four radio protocols, no external co-processor</b>

The tradeoff worth studying before you route your antenna: simultaneous multi-protocol on one die introduces RF interference that was previously partitioned to separate chips. The S31 datasheet's RF coexistence section is the first thing to read before finalising your layout.
