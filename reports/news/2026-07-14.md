<b>═ 🔥 TOP STORY ═</b>
Firmware security firm Binarly disclosed six new vulnerabilities (BRLY-2026-037 through BRLY-2026-042) in U-Boot's FIT signature-verification code — the exact routine that establishes Root of Trust on secure-boot chains across a huge share of embedded Linux devices. Two of the six allow code execution: BRLY-2026-037 is a null-pointer dereference in fdt_find_regions() that escalates to a stack-based buffer overflow when memory at address 0x0 is mapped, and BRLY-2026-038 is a stack buffer underflow in the same function (demonstrated on QEMU ARM) caused by a negative length value propagating from an unchecked fdt_get_name() call. All six bugs trigger while U-Boot is still parsing the FIT image — before signature verification completes — so a crafted image can hit these paths regardless of whether it's ultimately trusted. The vulnerable code dates back to U-Boot v2013.07, meaning over 50 stable releases and countless vendor forks are potentially affected, and the fixes missed the July 2026 U-Boot release because that release had already code-frozen in April. Why it matters: "secure boot" is only as strong as the code that runs before the signature check passes, and here the pre-check code path was the whole attack surface — a lesson that generalizes to any bootloader validating untrusted images, not just U-Boot.

<b>1. Post: Security wake-up — the code that runs before the signature check is the whole attack surface</b>
Six new bugs just landed in U-Boot's secure-boot verification code. Two of them let an attacker run code — before the signature check even finishes.

Binarly found that U-Boot's FIT image parser trusts data it hasn't verified yet. A null-pointer dereference in fdt_find_regions() escalates to a stack buffer overflow if memory at address 0x0 happens to be mapped. A separate bug uses a negative length value to walk a pointer backward until it overwrites the return address — demonstrated live on QEMU ARM.

The part that should bother anyone shipping secure boot: these paths run during FIT parsing, which happens before the cryptographic signature check completes. "Secure boot" only protects what happens after verification passes — if the code doing the verifying has exploitable bugs, the whole chain of trust is theater. The vulnerable code has been sitting in U-Boot since 2013, across 50+ stable releases.

If your boot chain relies on a bootloader parsing untrusted data before it validates that data, do you actually know what runs first?

<b>2. Post: Capability unlock — a camera that runs person detection with zero battery</b>
A camera just ran a person-detection model powered by nothing but ambient light. No battery. No supercap swap. Just photons in, inference out.

STMicroelectronics demoed this with the STM32U3B5/C5, a new ultra-low-power MCU built around a 0.65V near-threshold design and a dedicated HSP (hardware signal processor) accelerator. The Nucleo board plus camera ran entirely off photovoltaic harvesting while executing the inference workload.

The engineering unlock isn't "lower power MCU" — that's incremental. It's that the accelerator and the near-threshold voltage design were co-designed specifically so the harvested-energy budget could cover both sensing and inference, not just sleep-mode housekeeping. That reframes what "battery-free" edge AI can actually do beyond blinking an LED on a wake event.

<b>3. Post: Regulatory shift — the EU just set the actual bar for "CRA compliant"</b>
"Comply with the Cyber Resilience Act" has been a vague target for two years. On August 30, 2026, it stops being vague.

That's the delivery deadline for the first two of 41 harmonised standards the European Commission requested under mandate M/606 — specifically the two horizontal standards covering secure development and vulnerability handling. Roughly 26 more product-specific vertical standards follow by October 30, 2026.

Here's the part engineers should sit with: these standards are what "presumption of conformity" under the CRA will actually be graded against, not the regulation's text itself. Until they land, manufacturers have been building toward a moving target. Once they land, "secure development lifecycle" and "vulnerability handling process" become concrete, auditable checklists — not design philosophy.

If you're building anything with digital elements bound for the EU market, is your dev process already structured around a documented SDL, or would August 30 be the first time you'd have to prove it?

<b>4. Post: Integration leap — three radios, one RISC-V die, no external front-end</b>
Wi-Fi 6, Bluetooth LE, and Zigbee/Thread's 802.15.4 — on a single RISC-V microcontroller the size of a fingernail.

The M5Stack Stamp-C5 wraps Espressif's ESP32-C5 into a USB-C Feather-form board with up to 19 GPIO and battery support. The chip itself is what's interesting: dual-band Wi-Fi 6 (2.4 and 5GHz), BLE, and 802.15.4 all natively on one RISC-V die, no separate radio front-end needed per protocol.

The integration angle matters more than the board: multi-protocol wireless used to mean either a multi-chip BOM or a compromise on which radios you got. Collapsing three physically distinct RF stacks onto one die — while staying RISC-V, not licensed ARM cores — is the kind of silicon-level convergence that changes what a "small, cheap, multi-protocol node" costs to build.

<b>5. Post: Physics-meets-architecture — 340 nanoamps is what makes battery-free IoT real</b>
340 nanoamps. That's what it costs to keep a real-time clock running and 8KB of SRAM retained on e-peas' new ultra-low-power MCU, the EDMS105N.

The 24MHz Cortex-M0 draws 18µA/MHz in active mode and drops to 340nA in its deepest sleep state — paired with e-peas' own solar (AEM10941) and RF (AEM30940) harvesting PMICs and a purpose-built antenna, as one integrated reference stack rather than a chip you have to build a harvesting front-end around yourself.

The physics worth understanding: sleep-current numbers this low only matter if the harvesting front-end can actually refill the budget between duty cycles faster than leakage drains it. Publishing the MCU and both PMICs as one matched system is the tell that e-peas is optimizing the harvest-to-retention loop as a whole, not just chasing a headline sleep-current spec in isolation.
