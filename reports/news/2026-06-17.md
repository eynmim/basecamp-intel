<b>═ 🔥 TOP STORY ═</b>

CVE-2026-10635 — disclosed June 16 — is a use-after-free in Zephyr 4.4.0's Xtensa MMU page-table code, introduced in the most recent stable release (April 14, 2026). The root cause lives in arch/xtensa/core/ptables.c: the global active-domain list retains a stale pointer after k_mem_domain_deinit() removes the node, so any subsequent arch_mem_map() call traverses freed memory — producing at minimum a kernel-fatal NULL dereference, and under reuse conditions, arbitrary page-table corruption that breaks Xtensa hardware-enforced userspace isolation. The Xtensa LX6/LX7 architecture is the ESP32 family, meaning every ESP32 project running Zephyr 4.4.0 with CONFIG_USERSPACE + CONFIG_XTENSA_MMU is on a vulnerable build. The fix is on Zephyr main only — no 4.4.1 patch release exists as of June 17. Mitigation: build from main, or disable CONFIG_XTENSA_MMU if you do not rely on hardware-enforced userspace isolation.

<b>1. Post: [Security wake-up] — Zephyr 4.4.0, two months old, already has a use-after-free</b>
Zephyr 4.4.0 shipped April 14. CVE-2026-10635 was disclosed June 16 — 63 days later.

The bug is in arch/xtensa/core/ptables.c: when k_mem_domain_deinit() is called, the domain node is removed from the global list, but subsequent arch_mem_map() calls still traverse the stale pointer. Best case: a NULL dereference produces a fatal kernel exception. Worst case: if the freed domain struct has been reallocated, you get page-table corruption that breaks Xtensa MMU-enforced userspace isolation.

The architecture this affects is Xtensa LX6/LX7 — the ESP32 family. Every ESP32 project running Zephyr 4.4.0 with CONFIG_USERSPACE + CONFIG_XTENSA_MMU is vulnerable. The fix is on Zephyr main. No 4.4.1 patch exists yet.

The lesson isn't "Zephyr shipped a bug." Every codebase does. The lesson is: stable-release tracking without a security advisory feed leaves you discovering vulnerabilities weeks late. Subscribe to the Zephyr security mailing list.

<b>5. Post: [Physics-meets-architecture] — 18 µA/MHz. 340 nA sleep. How does a Cortex-M0 get there?</b>
e-peas's EDMS105N claims 18 µA/MHz active current and a 340 nA deep-sleep floor with RTC running and 8 kB SRAM retained. Those numbers are meaningfully lower than typical Cortex-M0+ devices from the main MCU vendors. The reason isn't a surprise architecture trick — it's co-design.

The EDMS105N was designed alongside e-peas's PMIC family. Power delivery, voltage-rail sequencing, and wake-up threshold logic are tuned together rather than mated post-design. The result: no general-purpose PMIC overshooting rail-up current, no regulator held on during sleep modes because a handshake signal lingers, no cold-start uncertainty. The AEM15820 harvesting PMIC cold-starts at 5 µW from 275 mV — meaning the whole system wakes from a nearly dark room.

e-peas demonstrated a fully batteryless system: AEM10941 (solar) + AEM30940 (RF) PMICs driving the EDMS105N through a complete sense-compute-transmit-sleep cycle on harvested energy alone.

The programming model this requires — fitting each wake cycle inside the harvested energy budget — is under-documented compared to conventional firmware. That's the interesting engineering challenge here.
