<b>═ 🔥 TOP STORY ═</b>
Espressif shipped ESP-IDF v6.1-beta1 on June 29, adding the first preview software support for the ESP32-S31 (dual-core RISC-V, Wi-Fi 6, BT 5.4, 802.15.4, Gigabit Ethernet) — the point where the chip becomes bring-up-ready rather than a paper announcement. The release also lands a genuinely new PSRAM feature: CONFIG_SPIRAM_ENC_EXEMPT lets firmware carve out a designated unencrypted PSRAM region via MALLOC_CAP_SPIRAM_NO_ENC, sidestepping the alignment constraints that full-PSRAM encryption normally imposes. A security fix hardens the JPEG decoder against malicious-image "bad picture" attacks, and Wi-Fi gains OWE-only SoftAP, DPP multi-configuration for station mode, and STA MAC-randomization (privacy extension) support. One breaking change to plan around: esp-mqtt has moved out of ESP-IDF core into the component manager, so existing projects need espressif/mqtt added explicitly as a dependency before upgrading. For anyone building on ESP32-S3 today, this beta is the first real look at what the S31 migration path and the v6.1 PSRAM/security model will actually require.

<b>1. Post: [Security wake-up] — A single UDP packet to a port nobody watches</b>
NBNS.begin() opens UDP/137 on your ESP32 board. Most firmware never touches it directly — it just runs, quietly, for name resolution.

CVE-2026-41429 is a stack buffer overflow sitting inside that parser. The vulnerable code trusts an attacker-controlled length field without checking it against the size of the destination buffer. Any unauthenticated device on the same network segment can trigger it with a single crafted packet.

The part worth internalizing: this isn't application logic anyone audits during a code review. It's a legacy protocol handler bundled into arduino-esp32, silently listening, that most developers never consciously chose to enable. Fixed in arduino-esp32 v3.3.8 — but only if you know to check.

What's running on your board's network stack that you didn't explicitly write?

<b>2. Post: [Physics-meets-architecture] — One PMIC, indoor light and direct sun, no switch</b>
Solar energy harvesting for IoT has always meant picking a lane: an amorphous-silicon power tree tuned for dim indoor light, or a crystalline-silicon tree tuned for direct outdoor sun. Different cells, different charge curves, different PMICs.

e-peas' AEM15820 collapses that choice into one chip. It auto-switches between an integrated low-power and high-power boost converter depending on where the input actually sits on the microwatt-to-watt range, cold-starting at just 5µW and 275mV.

The interesting design consequence isn't just BOM reduction — it's that the PMIC itself now makes a real-time power-topology decision your board doesn't have to. That used to be a firmware-level state machine; now it's silicon.

Anyone shipping a solar node that has to work both on a windowsill and outdoors — this removes a whole design branch.

<b>3. Post: [Tooling unlock] — ESP-IDF just gave PSRAM encryption an escape hatch</b>
Full PSRAM encryption on ESP32-S3 is good for security and bad for anyone doing DMA buffers, framebuffers, or shared memory with strict alignment needs — the encryption boundary imposes constraints that don't play well with those use cases.

ESP-IDF v6.1-beta1, released June 29, adds CONFIG_SPIRAM_ENC_EXEMPT: a way to carve out a designated unencrypted PSRAM region via a new memory capability flag, instead of an all-or-nothing choice.

The same beta quietly does something bigger for anyone tracking Espressif's next silicon generation — it adds the first preview support for the ESP32-S31, meaning firmware bring-up on that chip is now possible rather than theoretical.

One thing to check before you upgrade: esp-mqtt has moved out of ESP-IDF core into the component manager, so existing projects need an explicit dependency added or the build breaks.

https://github.com/espressif/esp-idf/releases/tag/v6.1-beta1

<b>4. Post: [Regulatory shift] — 72 days out, and the reporting platform doesn't exist yet</b>
September 11, 2026. That's when the EU Cyber Resilience Act's 24-hour vulnerability reporting duty activates for every product with digital elements sold in the EU.

As of June 29, the ENISA Single Reporting Platform — the actual tool manufacturers are supposed to report through — is not live. ENISA says instructions are coming "in the weeks ahead." The law's deadline isn't moving to match.

The practical read: if your compliance plan assumes a working government reporting portal will exist in time to test against, you're planning against an assumption, not a system. The manufacturers ahead of this are building internal detection and disclosure workflows now, independent of whether ENISA's platform ships on schedule.

72 days is not a lot of runway for a process you haven't tested yet.

<b>5. Post: [Capability unlock] — An MCU running AI inference with no battery in the design</b>
Battery-free sensing usually means dropping down to the simplest possible logic — a comparator, a counter, maybe a basic state machine. Real inference was assumed to need a battery.

STMicroelectronics' STM32U3B5/C5 pairs a Cortex-M33 core with an integrated hardware signal-processor accelerator, rated at 117 CoreMark/mW in active mode, aimed specifically at running lightweight AI/DSP workloads directly off harvested energy — no cell in the BOM.

The shift worth noting: this isn't a low-power MCU that can technically run a tiny model if you starve everything else. It's a part designed with the harvested-energy power budget as a first-class constraint on the accelerator itself, not an afterthought bolted onto a general-purpose low-power core.

Pair it with a PMIC like the AEM15820 above and the entire "battery or no battery" question stops being a hardware limitation and becomes a scheduling problem.
