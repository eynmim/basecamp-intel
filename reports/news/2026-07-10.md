The FatFs saga has moved from "watch" to "act": runZero's July 1 disclosure of seven CVEs in the tiny FAT/exFAT filesystem library vendored into ESP-IDF, STM32Cube, and Zephyr has now shipped with concrete CVE numbers, CVSS scores, and a working QEMU-based proof-of-concept exploit in a public companion repo. Only one of the seven bugs — CVE-2026-6684 — has an upstream fix, landed in FatFs R0.16; the headline bug, CVE-2026-6682 (CVSS 7.6, an integer overflow when mounting a FAT32 volume), and CVE-2026-6687 (an exFAT volume-label buffer overflow) remain unpatched more than a week after JPCERT/CC tried to reach FatFs's single maintainer. Because most embedded targets lack the memory protections of a phone or PC, runZero's blunt framing is "any physical access leads to a jailbreak" — a booby-trapped SD card, USB stick, or OTA image is enough. Why it matters: this hits the exact filesystem layer sitting under ESP-IDF and STM32Cube projects, and with exploit code now public, "watch" is no longer the right posture.

<b>1. Post: Security wake-up — the filesystem bug with a public exploit and no fix</b>
Six of seven CVEs. Still no patch. And now the exploit code is public.

runZero's FatFs disclosure started July 1 with seven vulnerabilities in the tiny filesystem library vendored into ESP-IDF, STM32Cube, and Zephyr. A week later, only one bug is fixed upstream — the headline integer overflow in FAT32 mounting (CVSS 7.6) and an exFAT buffer overflow are both still open, and JPCERT/CC still can't reach FatFs's single maintainer.

The part that should change your threat model: runZero shipped a working QEMU-based exploit in a public repo. Most embedded devices don't have the memory protections a phone or laptop has, so "any physical access leads to a jailbreak" isn't hyperbole — it's just how thin the layer between an SD card and code execution actually is.

If your firmware mounts FAT/exFAT media, is that now untrusted input in your threat model, or still an assumption?

<b>2. Post: Capability unlock — a microcontroller under $1 just got a neural accelerator</b>
90x faster inference. 120x less energy per inference. Under $1 in volume.

TI's MSPM0G5187 and AM13Ex MCUs integrate the TinyEngine NPU, a dedicated hardware accelerator that handles deep-learning inference operations that would otherwise burn through a general-purpose MCU's cycle budget and power envelope.

The number that matters isn't the chip — it's the price point. On-device AI inference used to mean picking a specialty accelerator or offloading to a bigger processor. When the accelerator shows up inside a sub-$1 general-purpose MCU, "does this product need edge AI" stops being a BOM-cost question and starts being a firmware-architecture question.

<b>3. Post: Tooling unlock — ESPHome quietly made every ESP32 project 34% faster</b>
No new hardware. No code changes. Just a config default flip, and ESP32 projects got ~34% faster.

ESPHome 2026.4.0 changed the default clock on ESP32, ESP32-S2, ESP32-S3, and ESP32-C5 targets from 160MHz to 240MHz — the chips could always run there, the default was just conservative. The same release added optional signature-required OTA updates, closing a real gap: unsigned OTA means anyone who can reach your update endpoint can flash arbitrary firmware.

The pattern worth noticing: a lot of "free performance" in embedded projects isn't algorithmic, it's just a conservative default nobody revisited once the silicon matured.

Worth an afternoon checking which of your own ESP32-S3 projects are still running at a default clock speed set years ago for a different, more conservative chip revision.

<b>4. Post: Integration leap — a license-free protocol just borrowed cellular-grade range</b>
DECT NR+ isn't cellular. It doesn't need a SIM. And it just showed up in a $-class USB dongle.

Norik Systems' new dongle runs on Nordic's nRF9151 SiP as a DECT NR+ node — a non-cellular 5G-standard protocol operating in the license-free 1.9GHz band, positioned for the same long-range, low-power sensor use cases LoRaWAN and NB-IoT currently own, minus the spectrum licensing or SIM/eSIM overhead NB-IoT requires.

The interesting part is architectural: DECT NR+ borrows 5G's standardized PHY/MAC design but runs it unlicensed, which is a genuinely different tradeoff point than either "unlicensed ISM band" (LoRaWAN) or "licensed cellular" (NB-IoT) — worth knowing before defaulting to LoRaWAN on the next long-range sensor design.

<b>5. Post: Regulatory shift — the EU just started the Cyber Resilience Act's clock for real</b>
December 2027 always felt far away. September 11, 2026 does not.

Most CRA coverage focuses on the main 2027 compliance deadline, but the Act's vulnerability-reporting duty — mandatory disclosure of actively exploited vulnerabilities and severe incidents — becomes binding in exactly two months, well ahead of everything else in the regulation.

For engineers, not just compliance teams, this matters: it means your product's vulnerability-handling process (how you receive reports, triage severity, and report upstream) needs to exist and function months before the rest of the CRA's security-by-design requirements land. Waiting for 2027 to start building that process is already late.

If you're building anything that ships into the EU, does your team have an actual vulnerability-reporting process today, or just an intention to build one?
