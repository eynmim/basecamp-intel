<b>═ 🔥 TOP STORY ═</b>

Two new Yealink SIP-T46U firmware CVEs (CVSS 8.0 each) were publicly disclosed yesterday — June 15 — both rooted in unsafe sprintf and strcpy calls with attacker-controlled uid and start_offset arguments inside the firmware OTA upgrade handler. CVE-2026-12220 overflows a stack buffer in mod_upgrade.SparePartsUpload; CVE-2026-12221 does the same in the /api/upgrade endpoint. Both are exploitable from the local network without authentication. These vulnerabilities land 87 days before EU Cyber Resilience Act Article 14 goes live on September 11, 2026 — the date when every embedded/IoT manufacturer shipping into the EU must report actively exploited firmware vulnerabilities to ENISA within 24 hours, under fines of up to €15 million. The engineering implication is blunt: OTA handlers are the highest-privilege network-accessible code path in most firmware, yet they routinely skip the input-length validation applied elsewhere. If you have an OTA update API and it uses sprintf with untrusted field lengths, check it today.

<b>4. Post: [Physics-meets-architecture] — 90x faster inference. Under $1. Here's why the numbers are real.</b>
TI's TinyEngine NPU claim — 90x lower inference latency, 120x lower energy per inference vs a general-purpose Cortex-M — sounds like marketing. The microarchitecture explains why it isn't.

A Cortex-M running a convolution layer cycles through fetch, decode, and execute for every multiply-accumulate operation, pulling weights through the cache hierarchy each time. A dedicated MAC array in the TinyEngine NPU operates on pre-staged SRAM with no instruction overhead — the bottleneck shifts from compute to memory bandwidth, which the NPU's local SRAM architecture is designed to minimise.

At under $1 in 1k-unit quantities (MSPM0G5187), this brings NPU-class inference — keyword spotting, gesture recognition, motor anomaly detection — into sensor nodes that previously couldn't afford the BOM.

TI CCStudio now ships 60+ pre-built models for TinyEngine targets. The quantise-flash workflow fits inside the IDE.

<b>5. Post: [Security wake-up] — sprintf in a firmware OTA handler. CVSS 8.0. Disclosed yesterday.</b>
CVE-2026-12220 and CVE-2026-12221, both disclosed June 15: stack-based buffer overflows in Yealink SIP-T46U firmware's OTA upgrade handler. Root cause — sprintf called with attacker-controlled uid and start_offset fields, no length bound. Both exploitable from the local network, no authentication required.

The specific location matters. This isn't in some obscure feature path. It's in the firmware update handler — the code that runs when the device decides to trust and install new firmware. If you overflow it, you have code execution before image validation even runs.

OTA handlers are the highest-privilege network-accessible path in most embedded firmware. They're also where developers often skip the defensive programming applied elsewhere, because "it's internal" or "only runs during updates."

Audit every sprintf in your OTA path. Use snprintf with explicit size. Add the upgrade endpoint to your fuzz scope in CI. The pattern that caused these CVEs is still endemic — because it's easy to write and the failure mode is invisible until someone sends a crafted packet.
