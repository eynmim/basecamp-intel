<b>2. Post: [Security wake-up] — Nation-state actors are exploiting firmware defaults, not zero-days</b>
The Russian GRU's 2026 SOHO router botnet was not built on exotic zero-days. It was built on default credentials and a system() call in a CGI handler.

The FBI IC3 advisory (PSA260407) describes the exploitation chain for TP-Link routers (CVE-2023-50224): default admin credentials combined with a command-injection flaw in the web management firmware, leading to full device compromise. Those routers were then used for credential theft and DNS hijacking against government and critical infrastructure targets — while owners had no idea.

For firmware engineers the lesson is structural. Web management interfaces in embedded firmware have used system() with user-supplied input for two decades, and the pattern is still common in new designs. CISA's response calls for three architecture changes, not patches: management interfaces restricted to LAN-side ports, enforced auto-update, and elimination of system() misuse in CGI code.

Nation-state adversaries are not breaking cryptography. They are exploiting the same firmware anti-patterns that appeared in security-training slides ten years ago.

<b>4. Post: [Integration leap] — Batteryless IoT nodes exist now. The unsolved part is firmware.</b>
The hardware for a node that never needs a battery replaced is available today, off the shelf.

e-peas' AEM15820 cold-starts at 5 µW from 275 mV — roughly a 5 cm² indoor PV cell under office lighting. Their companion EDMS105N MCU (Cortex-M0) draws 340 nA in deep sleep with the RTC running and 8 KB SRAM retained. Together they form a complete reference design for a batteryless sensor node that operates indoors without direct sunlight.

The unsolved engineering problem is in firmware. When power disappears — cloud cover, a light switched off — the node dies mid-execution. It restarts when light returns. But "restart" must mean "resume": same application state, same BLE bonding keys, same sensor context — not a cold boot from scratch. That requires deliberate firmware architecture: checkpointing state before each operation, NAND wear management for frequent writes, and a boot sequence that distinguishes a clean start from a power-loss resume.

The hardware crossed the threshold. The firmware patterns for graceful power-loss recovery at 340 nA budgets are still being standardised.
