<b>═ 🔥 TOP STORY ═</b>
ESET Research disclosed on July 14 that 11 Microsoft-signed Linux UEFI shim bootloaders (version 0.9 and below) contain a decade-old bug that fully bypasses Secure Boot. The root cause: a signed PE binary records its Authenticode signature length in two separate places — the PE header's data directory and the WIN_CERTIFICATE structure — and the affected shims read one location during the revocation check but the other during actual signature verification. By tampering with the WIN_CERTIFICATE structure of a second-stage loader, an attacker steers the revocation check toward the wrong bytes, so a certificate already revoked in dbx or MokListX quietly slips through and boots anyway. Two CVEs cover the case, CVE-2026-8863 and CVE-2026-10797, and because a vulnerable shim only needs to be present and trusted — not the active bootloader — an attacker can carry their own copy onto any UEFI machine that still trusts the Microsoft third-party UEFI CA 2011 certificate, regardless of installed OS; Microsoft revoked the known binaries in the June 9 Patch Tuesday update, but ESET says nobody knows how many more vulnerable shims are still out there since the underlying bug was fixed silently a decade ago without ever being flagged as a security issue. Why it matters: this is the same failure class as last week's U-Boot FIT-parsing bugs — a chain of trust is only as strong as the code that runs before the cryptographic check, and here that code has been shipping unnoticed since roughly 2016.

<b>1. Post: Security wake-up — a decade-old bugfix commit was actually a Secure Boot bypass</b>
Eleven Microsoft-signed UEFI shims have been quietly bypassing Secure Boot since around 2016. Nobody flagged it as a security issue until this week.

ESET found that these shims check two different fields for a signed binary's signature length — one during revocation checking, another during actual verification. Tamper with the right structure in a second-stage loader, and the revocation check reads the wrong bytes entirely. A certificate that's already been revoked boots anyway.

The unsettling part isn't the bug itself, it's how it surfaced: the fix was committed upstream a decade ago as a routine patch, with no CVE, no advisory, nothing that would have put it on a security team's radar. It just sat there, silently exploitable, on any UEFI system that trusts Microsoft's third-party CA — independent of which OS is actually installed.

If your threat model for "secure boot" stops at "is the signature check correct," are you also auditing the code that decides whether to trust the signature check?

<b>2. Post: Regulatory shift — Europe just turned "comply with the CRA" into a 24-hour clock</b>
Starting September 11, "we're working on CRA compliance" stops being a valid answer. It becomes: did you report within 24 hours?

That's the date the EU Cyber Resilience Act's incident-reporting obligation kicks in — manufacturers of any product with digital elements sold in the EU must send an early warning within 24 hours of learning about an actively exploited vulnerability, and a full notification within 72 hours. It's the first CRA deadline with real teeth, well before the 2027 full-compliance date everyone's been mentally filing this under.

Nordic Semiconductor's response is a tell: nRF Cloud now lets you upload a firmware SBOM and get automatic CVE matching across your entire connected fleet. That's not a nice-to-have anymore — it's the only realistic way to hit a 24-hour reporting window when you don't already know which of your shipped devices run the vulnerable library.

If you ship connected hardware into the EU, could you actually produce an accurate SBOM for your current firmware today?

<b>3. Post: Tooling unlock — compiling a neural network straight into C with zero runtime</b>
Most "AI on a microcontroller" stacks still ship a runtime. emmtrix's new tool skips it entirely — it compiles the ONNX model itself into plain, static C.

emx-onnx-cgen takes a trained ONNX model and generates C code with no dynamic memory allocation, no OS dependency, and no external inference engine linked in. Control flow and memory layout are fully determined at compile time — the same properties a MISRA or DO-178C reviewer wants to see in any other piece of firmware.

That's the real unlock: it's not about squeezing a model onto smaller silicon, it's about making the inference path auditable the same way the rest of a safety-relevant codebase already has to be. TFLite Micro solved "does it fit." This is aimed at "can you actually certify it."

Curious whether static-compiled inference like this becomes the default for safety-adjacent edge AI, or stays a niche next to interpreter-based runtimes.

Wi-Fi 6, Bluetooth 5.4, 802.15.4, and Gigabit Ethernet — on the same RISC-V die, with hardware JPEG and 2D-graphics acceleration thrown in.

Espressif's ESP32-S31 is a dual-core RISC-V SoC at up to 320MHz with 512KB SRAM and support for 250MHz DDR PSRAM. What stands out isn't any single spec — it's that a wired Gigabit MAC now sits alongside a full wireless radio stack and a display/camera pipeline on one part, in a product line that started as "just" a Wi-Fi microcontroller.
