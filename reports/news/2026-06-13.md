<b>═ 🔥 TOP STORY ═</b>

Security researchers at Cyloq published a full technical write-up this week on CVE-2026-0714 and its failed remediation CVE-2026-0715 — the first public documentation of a TPM2_NV_Read-path disk-key extraction on an embedded Linux ICS device. The target is Moxa's UC-1222A Secure Edition industrial computer, which uses LUKS full-disk encryption with the disk-encryption key stored in a discrete TPM 2.0 chip accessed over SPI. Moxa's patch for CVE-2026-0714 added TPM2 parameter encryption — the right countermeasure in principle — but omitted correct authorization-session binding, so the channel is encrypted while the session itself is unauthenticated: the LUKS key still crosses the SPI bus in a form trivially extractable by any attacker who opens the device and clips onto the SPI bus during boot. Remote exploitation is not possible and physical access is invasive, but for ICS hardware deployed in partially accessible locations (substations, outdoor enclosures) physical-access assumptions are weaker than in IT settings. The real lesson for embedded engineers is architectural: LUKS-on-TPM is not a plug-and-play secure pattern — encrypting the transport without authenticating the session is security theater, and many production embedded Linux designs likely replicate the same omission.

<b>1. Post: Security wake-up — LUKS + TPM is only as strong as the session auth binding</b>
I assumed LUKS-on-TPM was a solid security architecture for embedded Linux. CVE-2026-0714 documented exactly how it fails.

Moxa's UC-1222A industrial computer uses full-disk LUKS encryption with the key sealed in a discrete TPM 2.0 chip over SPI. Researchers sniffed the bus during boot and the LUKS key came out in plaintext.

The attack targets TPM2_NV_Read — not the more commonly studied TPM2_Unseal path. Moxa patched it (CVE-2026-0715) by enabling TPM2 parameter encryption. What the patch got wrong: it enabled encryption without correctly configuring the authorization session, so the channel is formally encrypted while the session is unauthenticated. The key still crosses the wire in extractable form.

The practical lesson for any embedded Linux design using LUKS + TPM: parameter encryption is necessary but not sufficient. You need correct authorization session binding — verifying the session, not just encrypting the transport. Many production designs probably replicate this omission.
