Three Linux kernel Wi-Fi CVEs — CVE-2026-46125, CVE-2026-46136, and CVE-2026-46152 — were published May 28, 2026 by the kernel security team, all targeting the generic mac80211 stack and the mt76 MediaTek driver used across a wide class of embedded Linux gateways, routers, and IoT platforms. CVE-2026-46136 is a buffer length underflow in the mt76 driver's country-location-control (CLC) handler: a malformed beacon frame triggers an unbounded read loop that crashes the Wi-Fi interface, no authentication required, adjacent-network attacker only. CVE-2026-46152 is a race condition in mac80211's Fast-RX path — a static variable shared across concurrent receive threads allows a timed burst of frames to trigger kernel panic or silent traffic manipulation including DNS hijacking on unencrypted streams. CVE-2026-46125 is a use-after-free in the Wi-Fi 7 mac80211 path, affecting embedded platforms with 802.11be radios. All three are backported to stable kernel branches (6.1, 6.6, 6.12) as of June 1; any Yocto, Buildroot, or OpenWrt image built from kernel sources before that date ships all three vulnerabilities on any Wi-Fi-enabled embedded Linux platform.

<b>1. Post: Security wake-up — mac80211 Fast-RX silent traffic hijack</b>
A static variable in the Linux Wi-Fi receive path can silently redirect your DNS traffic — no authentication, no user interaction, just a timed burst of 802.11 frames.

CVE-2026-46152 is a race condition in ieee80211_invoke_fast_rx(), the performance-optimised receive path in Linux mac80211. A static variable is shared across concurrent receive threads, so two simultaneous paths overwrite each other's per-call state — causing frame confusion between mesh and 802.3 receive paths.

The unusual part: exploitation doesn't crash loudly. It can silently redirect unencrypted traffic — DNS, HTTP — with no application-layer disruption visible in logs. It affects every cfg80211/mac80211-based driver across all vendors, not one specific chipset.

If you're running embedded Linux gateways on any Wi-Fi platform, your only fix is a post-June 1 kernel build on stable branches 6.1, 6.6, or 6.12. Worth checking your Yocto or Buildroot pinned kernel version today.
