<b>═ 🔥 TOP STORY ═</b>
Researcher Jaeyoung Chung disclosed "Bad Epoll" (CVE-2026-46242) this week, a use-after-free race condition in the Linux kernel's epoll subsystem that lets any unprivileged local user escalate straight to root. The bug traces back to a 2023 change in the epoll code, affects kernel v6.4 and later, and hits desktops, servers, and Android alike — Chung's exploit succeeds 99% of the time despite a race window only six CPU instructions wide, and it beat AI-based kernel auditing tools that had scanned the same code path. Epoll cannot be disabled, so there is no workaround; the only fix is pulling upstream commit a6dc643c6931. No in-the-wild exploitation has been confirmed yet and it isn't on CISA's KEV list, but any embedded Linux deployment — Yocto builds, Android-based gateways, industrial HMIs — running kernel 6.4+ inherits this until patched. For anyone running full Linux rather than an RTOS on their boards, this is a today problem, not a someday one.

<b>1. Post: Security wake-up — the six-instruction race that beats AI code review</b>
A security researcher just found a Linux kernel bug that AI-based auditing tools scanned right past — and it gives root to anyone with a local shell, 99% of the time.

"Bad Epoll" (CVE-2026-46242) is a use-after-free in the kernel's epoll subsystem: two code paths try to clean up the same object at once, one frees the memory while the other is still writing to it. The race window is six CPU instructions wide. Jaeyoung Chung built a working exploit anyway.

The part that should worry embedded/Linux engineers specifically: epoll can't be disabled, there's no config flag to turn it off, so there's no workaround — only the upstream patch. It hits kernel v6.4+ across desktops, servers, and Android, which means any Yocto build or Android-based gateway on that kernel range inherits it until patched.

If you're running full Linux rather than an RTOS on your boards, when did you last check your kernel version against the CVE list?

<b>2. Post: Regulatory shift — the compliance standard isn't finished yet</b>
Manufacturers are racing to comply with a cybersecurity standard that, as of this week, still doesn't officially exist in its final form.

The EU Cyber Resilience Act leans on IEC 62443 as its de facto technical baseline for connected products, but the harmonized version manufacturers actually need to point to for conformity isn't expected to land until the end of 2026. That's on top of the CRA's Article 14 24-hour vulnerability-reporting duty, which starts September 11 regardless.

The practical squeeze: full CRA conformity is due December 2027, meaning teams could have less than 12 months between the harmonized standard being finalized and the deadline to prove they meet it. Building your security documentation against draft guidance now, rather than waiting for the final text, isn't optional anymore — it's the only way the timeline works.

<b>3. Post: Integration leap — one chip, two Wi-Fi bands, three radios</b>
Dual-band Wi-Fi 6 finally showed up on a sub-$15 microcontroller board — and it's paired with Bluetooth LE and 802.15.4 on the same die.

The Waveshare ESP32-C5-LCD-1.47 is the first board built around Espressif's ESP32-C5, and it's a genuine step up from the ESP32-C6 it resembles: instead of 2.4 GHz-only Wi-Fi 6, the C5 does both 2.4 and 5 GHz, alongside Bluetooth 5 LE and 802.15.4 for Zigbee/Thread — all on one RISC-V core.

The engineering angle that matters here isn't the radio count, it's coexistence. Three radios sharing one antenna feed and one RF front-end means real firmware work goes into time-slicing and interference management that a single-radio ESP32-S3 design never has to think about — and now that complexity is available to hobbyists for the price of a board, not just Espressif's own reference designs.

<b>4. Post: Tooling unlock — when your RTOS vendor walks away</b>
Arm just stopped shipping fixes for Mbed OS. Not "deprecated with a migration guide" — actually stopped: no more CI, no more patches, codebase archived this month.

Mbed TLS survives separately under Trusted Firmware, so the crypto layer is fine. But the RTOS and the online build tools that a lot of hobbyist and early-career embedded projects grew up on are done. The only forward path from Arm's side is a community fork, Mbed CE.

The real lesson isn't about Mbed specifically — it's that RTOS choice is a build-vs-buy decision with a vendor-risk dimension people underweight. Zephyr's model (foundation-governed, not single-vendor) is exactly the hedge against this scenario, which is probably why so much of the industry has been quietly migrating toward it over the last two years.

<b>5. Post: Capability unlock — your LoRaWAN sensor can now talk to a satellite, unmodified</b>
Europe just cleared the spectrum for an off-the-shelf LoRaWAN sensor to talk directly to a satellite — no new radio, no firmware change, no new hardware revision.

The Electronic Communications Committee approved Decision ECC/DEC(25)02, opening the 862–870 MHz short-range-device band for direct satellite-to-device LoRaWAN links. Existing commercial LoRaWAN end devices can already communicate with LEO/GEO constellations operated by LoRa Alliance members like Lacuna Space and Plan-S.

What's easy to miss: this isn't a new protocol or a hardware upgrade path — it's a regulatory unlock for equipment that's already shipping. A soil-moisture sensor or asset tracker designed for terrestrial gateway coverage can now fall back to satellite in a dead zone, using the same radio it already has, if the network operator supports it.

Rural and remote-asset IoT just got a coverage floor it didn't have before — where do you see this mattering first, agriculture, logistics, or utilities?
