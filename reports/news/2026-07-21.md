<b>═ 🔥 TOP STORY ═</b>
Security researcher HD Moore (runZero) disclosed seven CVEs in ChaN's FatFs, the tiny FAT/exFAT filesystem library embedded in almost every RTOS an embedded engineer touches — including Espressif ESP-IDF, STMicroelectronics STM32Cube, Zephyr, FreeRTOS-adjacent stacks, MicroPython, ArduPilot, RT-Thread, Mbed OS, and Samsung TizenRT. The worst, CVE-2026-6682 (CVSS 7.6), is an integer overflow in FAT32 mount arithmetic that lets attacker-controlled file-size metadata on a crafted USB stick, SD card, or firmware update image trigger heap/stack corruption and code execution. FatFs has no CVE history, no security mailing list, and no coordinated disclosure process, so of the seven bugs only one (a GPT denial-of-service, CVE-2026-6684) has a fix — the other six, including the code-execution bug, remain unpatched, and each downstream project (ESP-IDF, STM32Cube, Zephyr, etc.) has to patch independently. Why it matters: this is a "silent dependency" problem — FatFs ships invisibly inside SDKs engineers never audit directly, and the bugs were found via an LLM-assisted re-read of code last reviewed in 2017, a preview of how much latent risk sits in decade-old embedded libraries nobody revisits.

<b>1. Post: Security wake-up — code from 2017 just failed an AI-assisted audit, seven times</b>
FatFs has no CVE history, no security mailing list, and sits invisibly inside nearly every embedded SDK you've ever used. That silence just ended.

A researcher used an LLM to re-read FatFs code last reviewed in 2017 and came out with seven bugs — including CVE-2026-6682, an integer overflow in FAT32 mount arithmetic that can lead to code execution from a crafted SD card or USB stick. It ships inside ESP-IDF, STM32Cube, Zephyr, MicroPython, ArduPilot, and more.

The uncomfortable part: FatFs has no coordinated disclosure process, so each downstream project has to patch independently, and six of the seven bugs — including the code-execution one — are still unpatched.

If a library has been "stable" since 2017 with zero security review, is that a track record or just a gap nobody's checked?

<b>2. Post: Capability unlock — always-on AI inference from a coin cell, not a battery pack</b>
Always-on inference has always meant a power budget you had to justify. This chip is trying to make that budget disappear.

Ambient Scientific's GPX10 Pro pairs an Arm Cortex-M4F with ten programmable AI cores delivering up to 512 GOPS peak, claiming sub-100µW operation and up to 100x better power/performance/area than a conventional 32-bit MCU running the same CNN/RNN workload.

The interesting design shift: this isn't "duty-cycle your inference to save power," it's architecture built so always-on stops being the expensive choice. That changes what's worth putting a model on in the first place — sensors that previously had to wake, infer, and sleep can just stay on.

Would you design a product around always-on inference if the power budget genuinely stopped being the constraint?

<b>3. Post: Integration leap — your ESP32 board can now speak the same protocol as next-gen smart locks</b>
Access control used to be its own silo — separate silicon, separate certification, separate vendor ecosystem from the rest of home/building IoT. Espressif just folded it into the boards you already use.

The new esp-aliro SDK brings the CSA's Aliro standard — a Matter extension purpose-built for smart locks and access control — to ESP32-C (Wi-Fi) and ESP32-H (Thread) SoCs, paired with a third-party NFC chip. This first release covers Aliro-over-NFC, with BLE and UWB transports planned.

What stands out: Aliro isn't a competing standard bolted alongside Matter, it's an extension of it — so a Thread-based ESP32 project already speaking Matter gets access control as an incremental SDK addition, not a redesign.

Anyone already have a Thread/Matter project that's one NFC chip away from becoming a lock?

<b>4. Post: Regulatory shift — the EU just gave embedded manufacturers a 24-hour clock</b>
Most security regulations set deadlines in months. This one starts counting in hours.

Under the EU Cyber Resilience Act, from September 11, 2026, manufacturers of in-scope connected products must report actively-exploited vulnerabilities and severe security incidents to ENISA and national authorities within 24 hours of becoming aware — more than a year before the CRA's full product-security obligations even apply.

The part worth sitting with: 24 hours means the reporting process has to exist and be rehearsed before an incident happens, not built in response to one. That's an internal process requirement disguised as a disclosure rule.

If your team shipped an EU-market connected product today, could you actually hit that 24-hour window?

<b>5. Post: Tooling unlock — a changelog line just turned off Secure Boot, on purpose</b>
Not every scary-sounding SDK change is a downgrade. Sometimes disabling a security feature is the fix.

ESP-IDF v5.5.5 disables ECDSA-based Secure Boot V2 on the ESP32-H2, ESP32-C5, and ESP32-P4 after Espressif found a vulnerability in that specific signing flow — meaning the "secure" option was quietly less secure than turning it off until it's properly fixed.

The takeaway for anyone shipping on those three chips: check your SDK version and secure-boot config now, don't assume "we enabled Secure Boot" is still true after an update — verify it against the current release notes every time.

Anyone actively running Secure Boot V2 on an H2, C5, or P4 design who hasn't checked this yet?
