<b>═ 🔥 TOP STORY ═</b>
Firmware security firm Binarly disclosed six new flaws in U-Boot's FIT (Flattened Image Tree) signature-verification code, tracked as BRLY-2026-037 through 042. Two allow code execution and four cause denial-of-service, and all six trigger while U-Boot is still parsing an untrusted image — before signature validation ever completes — so a malformed image can hijack or crash the boot chain on the very check meant to stop it. Most of the vulnerable code traces back to U-Boot v2013.07, meaning it has shipped unnoticed across more than 50 stable releases and countless vendor forks running on routers, cameras, and embedded controllers. The fixes were merged upstream in June, but the v2026.07 release had already code-frozen in April and shipped without them — the next release carrying the patches, v2026.10, isn't due until October 5. Why it matters: this is the second boot-chain trust bug to surface in two weeks (after last week's UEFI shim Secure Boot bypass) — a reminder that the code parsing an image before the cryptographic check runs is just as much attack surface as the check itself, and in U-Boot's case there's no backport to grab for over two months.

<b>1. Post: Security wake-up — the boot chain bug that hides until after your signature check runs</b>
Six new U-Boot bugs were just disclosed. Two let an attacker run code, and all six happen before the bootloader ever checks the signature that's supposed to stop them.

Binarly traced the root cause to U-Boot's FIT image parser: it calls fdt_get_name on untrusted data, and on a malformed image that lookup can return a null pointer and a negative length — both consumed downstream without a check. Most of the vulnerable code has been sitting in U-Boot since v2013.07, across more than 50 releases and every vendor fork built on top.

The timing is the part that stings. The fixes landed upstream in June, but U-Boot's v2026.07 release had already code-frozen in April. It shipped without them. The next release carrying the patches, v2026.10, isn't out until October — so for routers, cameras, and embedded controllers running plain upstream U-Boot, this is a known, disclosed, unpatched gap for roughly three months.

If your secure-boot threat model only covers "is the signature check correct," when did you last audit the code that runs before that check even starts?

<b>2. Post: Physics-meets-architecture — an AI chip that never executes an instruction</b>
Most "AI on a microcontroller" story is about shrinking a model to fit an instruction set. This chip skips the instruction set for the AI part entirely.

Ambient Scientific's GPX10 Pro maps a neural network's matrix-multiply and activation math directly onto ten analog in-memory compute blocks — no fetch-decode-execute loop for the AI workload, just current flowing through circuits shaped like the math. The result: 512 GOPS at 100MHz, over 7 TOPS per watt, and under 100 microwatts total draw, with a conventional Cortex-M4F alongside it for everything that isn't a neural net.

The interesting part isn't the headline efficiency number, it's what it implies architecturally: when compute is physically the computation instead of a program describing it, "years of always-on inference on a coin cell" stops being a stretch goal and becomes the baseline. That's a different design space than squeezing a quantized model onto an M55 with Helium instructions — it's genuinely a different way to build the silicon.

Digital NPUs or analog in-memory compute — which one do you think wins the always-on, coin-cell-powered inference category?

Wanting an FPGA next to your MCU used to mean two dev kits, two toolchains, and a ribbon cable between them. Forgix puts both on one Teensy-sized board for $50.

<b>4. Post: Tooling unlock — real-time instruction trace lands on cheap RISC-V MCUs</b>
Hardware trace debugging has mostly been an Arm Cortex-M privilege. SEGGER just extended it to RISC-V, on the same probes people already own.

J-Link and J-Trace now support Andes Trace, a hybrid of the N-Trace standard with an efficient E-Trace encoder. Practically, that means RISC-V parts — including the ESP32-C and ESP32-H families — can get real-time instruction trace on the same debug hardware Cortex-M developers use daily, instead of the printf-and-pray debugging that's often been the RISC-V default.

The gap this closes matters more than it sounds: trace-level visibility into timing bugs, ISR overruns, and race conditions has been a genuine capability tax on choosing RISC-V over Arm for anything beyond hobby projects. Closing it removes one more reason to default to Cortex-M out of tooling inertia rather than architecture merit.

If RISC-V debugging now matches Cortex-M's, what's left as the deciding factor for your next MCU choice?

<b>5. Post: Regulatory shift — the US just started building its own version of the EU's CRA</b>
The EU's Cyber Resilience Act isn't the only IoT security labeling regime anymore. The US now has one too, and it's about to become mandatory.

The FCC's U.S. Cyber Trust Mark program, now run by the ioXt Alliance after its previous administrator withdrew, becomes mandatory for anyone supplying consumer IoT products to the US federal government starting January 4, 2027. Meanwhile the EU Commission just published its first living FAQ on CRA implementation, filling in the details ahead of the September 11 vulnerability-reporting deadline.

For anyone building connected hardware with an eye on more than one market, that's two overlapping-but-different compliance tracks arriving on similar timelines: one built around mandatory incident reporting (CRA), one built around a voluntary-but-soon-federally-required label (Cyber Trust Mark). Neither maps cleanly onto the other's requirements.

If you're designing for both EU and US markets, are you tracking these as one compliance program or two?
