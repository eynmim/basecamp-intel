<b>═ 🔥 TOP STORY ═</b>
The Zephyr Project's security team published four separate GitHub Security Advisories in three days (June 28–30), an unusually tight cluster that reads like the output of a dedicated audit or fuzzing sweep rather than routine bug triage. The two most severe: CVE-2026-10646, a use-after-return in zsock_getaddrinfo() where a stack-allocated resolver-state object survives a DNS timeout retry and can be corrupted by a spoofed response (CVSS 7.4); and CVE-2026-10593, a NULL-pointer dereference in the Bluetooth LE Audio BAP unicast client reachable from a malicious remote ASCS server via a crafted GATT notification (CVSS 6.5). Both affect the still-supported 4.0–4.4 branches, and the fix isn't landing in a point release — it's queued for Zephyr 4.5.0, targeted for October 19, over three months out. For anyone shipping Zephyr-based BLE Audio or network-connected devices, that's a long exposure window with no official backport timeline yet announced, meaning affected teams need to either cherry-pick the advisory patches themselves or restrict the vulnerable code paths in the interim.

• <b>CVE-2026-10646: Zephyr zsock_getaddrinfo() Use-After-Return</b> [High] — A resolver-state struct is passed to the async DNS resolver by stack pointer; on timeout the code retries without cancelling the prior query, leaving a stale callback pointing at out-of-scope stack memory that a spoofed DNS response can corrupt. Affects Zephyr 4.0.0–4.4.0, fixed in 4.5.0 (CVSS 7.4). <a href="https://github.com/zephyrproject-rtos/zephyr/security/advisories/GHSA-h752-vhmf-29w6">Source →</a>

Fixed in Zephyr 4.5.0 (CVSS 6.5). <a href="https://github.com/zephyrproject-rtos/zephyr/security/advisories/GHSA-22q8-m94g-2pwh">Source →</a>

<b>1. Post: [Security wake-up] — Four Zephyr CVEs in three days, one fix in three and a half months</b>
When a security team ships four advisories in three days, that's not bad luck — it's usually the output of a dedicated audit pass finding what was always there.

Zephyr's June 28–30 batch includes a use-after-return in the DNS resolver (a stack-allocated struct outlives its timeout retry, and a spoofed response can corrupt it) and a NULL dereference in the Bluetooth LE Audio unicast client, reachable from a malicious remote server via a crafted GATT notification.

The part that actually matters for anyone shipping on the affected 4.0–4.4 branches: the real fix isn't a point release. It's bundled into Zephyr 4.5.0, targeted for October 19 — over three months away. If you're exposed, you're patching manually or waiting.

Worth checking your Zephyr branch against the advisory list today.

<b>2. Post: [Physics-meets-architecture] — 0.65V and 117 CoreMark per milliwatt</b>
Most MCUs give up computing entirely once you drop voltage into "energy harvesting" territory. ST's new STM32U3B5/U3C5 keeps running.

The trick is a near-threshold design — running the Cortex-M33 core close to the transistor's actual switching threshold instead of comfortably above it. That gets you 117 CoreMark/mW in active mode, and the chip still operates down to 0.65V. There's also a dedicated hardware signal-processing accelerator handling DSP/AI workloads, efficient enough that battery-free operation off harvested energy is a real target, not a marketing footnote.

The interesting trade-off: near-threshold operation usually means more sensitivity to process/voltage/temperature variation. Whether that shows up as real-world reliability headaches at 0.65V is the question I'd want a datasheet appendix on before committing a design to it.

Anyone running a harvested-power design near this voltage floor — what's your PVT margin been like in practice?

<b>3. Post: [Capability unlock] — Bluetooth ranging just got a hardware shortcut</b>
Channel Sounding turns a BLE radio into a distance sensor — useful for anti-relay car keys, precise asset tracking, anything where "which room" isn't good enough.

Bluetooth Core 6.3's new "Inline PCT Transfer" changes how the reflector side reports its phase data: instead of collecting phase-and-tone measurements and shipping them as a separate report, the reflector writes phase-aligned tone data straight to hardware. Less overhead per ranging exchange, and PHY-specific RTT accuracy reporting now lets a stack reason about accuracy differently across PHYs in the same session.

None of this is exotic radio physics — it's protocol-level plumbing that removes a round trip. But that's exactly the kind of change that turns "ranging works in the lab" into "ranging is fast enough to ship."

Curious whether NimBLE or Zephyr's BLE host picks this up before end of year.

<b>4. Post: [Price-floor shift] — A closed-loop motor controller MCU for two cents</b>
Motor control used to mean a general-purpose MCU plus a lot of your own FOC (field-oriented control) math running on the main core, competing for cycles with everything else.

Fortior's new FU75xx puts a 32-bit RISC-V core next to a dedicated second-generation "Motor Engine" — a separate core with hardware CORDIC/FOC modules that runs the control-loop math in as fast as 5 microseconds, decoupled entirely from whatever the RISC-V side is doing. Nine SKUs cover the range, and the smallest variant reportedly sells for around two cents a unit at volume.

That's not a rounding error in a BOM — it's the kind of price point that makes a dedicated FOC core the default choice for BLDC/PMSM drives instead of a nice-to-have, and it's already shipping in Banana Pi's new open-source servo controller board.

Worth watching whether this pulls dedicated motor-control silicon further down into hobbyist/prototyping price territory.

<b>5. Post: [Regulatory shift] — Italy just changed how the work-permit line forms</b>
For years, Italy's Decreto Flussi work-permit quotas were a "click day" — thousands of applications fired at a portal in the same second, and speed decided who got a slot.

The newly approved 2026–2028 decree changes the mechanism: 164,850 total entries (76,850 non-seasonal), but allocation now happens through a post-application provincial process instead of raw click-day speed, paired with a new statutory 30-day deadline for the nulla osta itself to be issued.

For engineers navigating non-EU-to-EU mobility, that's a meaningfully different planning problem — you're no longer optimizing for reflexes at midnight, you're optimizing for how your application gets processed provincially, and you have an actual deadline to hold the system to.

If you're mapping out an EU work-permit timeline, this is worth reading past the headline number.
